# Grizzly Messenger — что нужно для легального запуска в РФ

> **Это не юридическая консультация.** Практический разбор на основе открытых
> источников (149-ФЗ, 152-ФЗ, 241-ФЗ, 374-ФЗ, ПП-313, ПП-1279, КоАП) и
> фактического состояния кода на 2026-04-22. Перед любым действием — адвокат.
>
> Связанный документ: [LEGAL_RESPONSE.md](LEGAL_RESPONSE.md) — что делать, когда
> запросы уже пришли. Эта памятка — про то, как пройти до запуска.

---

## TL;DR

Изначальная оценка («ИП + нотификация ФСБ + РКН + хостинг в РФ») была
**неполной**. По факту кода проект попадает под **три регуляторных режима
одновременно**, и часть из них требует договорных отношений с третьими
сторонами, которые получаются небыстро и небесплатно.

| Что я сказал раньше | Что на самом деле |
|---|---|
| Нотификация ФСБ | ✅ Да, и её достаточно для крипты — это верно |
| ИП, УСН 6% | ✅ Да, но для 241-ФЗ может понадобиться **ООО** |
| Реестр ОРИ «можно отложить» | ❌ Не для мессенджера. 241-ФЗ привязывает старт работы к идентификации |
| OTP через Twilio/Firebase | ❌ В РФ нужен договор с **российским оператором связи** об идентификации |
| Хранение 1 год метаданных | ❌ По 374-ФЗ: **метаданные 3 года, контент 6 мес** |
| Запуск за 1.5 месяца | ⚠️ Реалистично 3–6 месяцев при параллельной работе |

## 1. Что обнаружено в коде (фактическая поверхность регулирования)

### 1.1 Криптография

Файл: [app/src/main/java/com/secure/messenger/utils/CryptoManager.kt](app/src/main/java/com/secure/messenger/utils/CryptoManager.kt)

- **X25519 ECDH** (BouncyCastle) — согласование ключа
- **HKDF-SHA256** — деривация ключа сообщения (salt = messageId)
- **AES-256-GCM** — шифрование контента (128-bit auth tag)
- **PBKDF2-HMAC-SHA256, 100 000 итераций** — деривация ключа из пароля + телефона
  (`deriveKeyPairFromPassword`), чтобы одним паролем открывать переписку с любого
  устройства
- WebRTC: **DTLS-SRTP** (из коробки, AES-128-CM)
- Транспорт: **TLS/WSS**, cleartext запрещён `network_security_config.xml`

**Регуляторная квалификация**: всё это — **стандартная массовая криптография**,
используются общепринятые алгоритмы и длины ключей, публично опубликованные.
Попадает в **нотификационный режим** по Положению о нотификации шифровальных
средств ЕАЭС (категории 5.А.2, 5.D.2 — «массовое ПО для конечного пользователя»).

**Лицензия ФСБ на разработку/распространение СКЗИ не требуется**, пока:
- не пишете свои алгоритмы;
- не позиционируете продукт как СКЗИ;
- не хранитесь в централизованном виде ключей пользователей.

**⚠️ Потенциальный риск**: PBKDF2 с пользовательским паролем — если пароль
слабый, ключ брутфорсится офлайн. Для нотификации это не критично, для
пиара/доверия — да. 100k итераций в 2026 году маловато, стандарт OWASP —
600 000+ для PBKDF2-SHA256 или переход на Argon2id.

### 1.2 Серверная архитектура

Стек (из [server/go.mod](server/go.mod)):
- Go 1.23, chi, PostgreSQL (pgx), WebSocket (gorilla), JWT
- Docker-compose, TURN-сервер (coturn, `turnserver.conf`)
- Админка: [server/internal/admin/](server/internal/admin/) с UI на
  `https://grizzly-messenger.ru/admin/`

Найденные функции админки (релевантные для регулятора):
- `DeleteMessage` — админ **удаляет чужие сообщения** и рассылает
  `message_deleted` всем участникам чата через WebSocket
- `BotUserID = "00000000-0000-0000-0000-000000000001"` — системный бот
- **Массовая рассылка**: админ отправляет сообщение от бота **всем
  пользователям** (создаёт direct-чат с каждым, инсертит `SYSTEM`-сообщение)
- Редактирование уже разосланных системных сообщений
- Регистрация версий APK через `POST /admin/versions`, самообновление в обход
  сторов (`/v1/app/update`)

### 1.3 Идентификация

- Регистрация по номеру телефона + OTP (SMS)
- Управление сессиями с геолокацией (страна, город) и IP
- `fcm_token` для пушей
- Поле `users.phone` хранится **в открытом виде** (в `LEGAL_RESPONSE.md`
  запланирован переход на хеши — пока не сделан)

### 1.4 Доменное имя и платформа распространения

- **grizzly-messenger.ru** — домен в зоне .RU, регистратор требует паспортные
  данные физлица или реквизиты юрлица (публично в WHOIS через дней 60)
- APK раздаётся напрямую с сервера, не через сторы

## 2. Какие режимы регулирования применяются

### 2.1 🔴 241-ФЗ «Об организации обмена мгновенными сообщениями» (главное)

Это **отдельный закон именно про мессенджеры**, 2017 года, с поправками 2020–2024.

**Кто подпадает**: организатор сервиса обмена мгновенными сообщениями. Grizzly
подпадает полностью.

**Что требует**:

1. **Идентификация пользователя по абонентскому номеру** — не просто OTP, а
   именно сверка с данными оператора связи.
2. **Договор об идентификации с российским оператором связи** (МТС, Билайн,
   МегаФон, Т2 или МВНО). Это платный договор, процедура согласования — от
   2 до 6 месяцев.
3. **Ограничение функционала для неидентифицированных** — либо не пускать
   вообще, либо ограничивать (например, без отправки сообщений).
4. **Обязанность не допускать** передачу сообщений, содержащих запрещённую
   информацию (из реестра запрещённых сайтов/материалов РКН).
5. **Обязанность обеспечить возможность отказа пользователя** от получения
   массовых рассылок (привет, `GrizzlyBot`).
6. **Запрет массовых рассылок без согласия** — до рассылки от бота нужно
   получить явное согласие пользователя.

**Санкции (КоАП 13.39.1, 13.39.2)**:
- Отсутствие идентификации: до 1 млн ₽ на юрлицо.
- Массовая рассылка без согласия: до 500 000 ₽ за каждое нарушение.
- Передача запрещённой информации: блокировка по решению Генпрокуратуры.

**Что это значит для Grizzly**:
- **OTP через Twilio / Firebase — не годится**, нужен договор с РФ-оператором.
- **Grizzly Bot в его нынешнем виде** (админ жмёт кнопку — всем приходит
  сообщение) — **нарушение**, если нет opt-in-согласия. В [CHANGELOG.md:45](CHANGELOG.md#L45)
  это описано как фича — её надо либо перестраивать, либо делать явный
  onboarding-чекбокс «согласен получать уведомления от сервиса».

### 2.2 🔴 149-ФЗ, ст. 10.1 — реестр ОРИ

**Кто подпадает**: Grizzly как сервис обмена сообщениями — да, и 241-ФЗ прямо
ссылается на то, что мессенджеры **одновременно** являются ОРИ.

**Что требует**:

1. **Внесение в реестр** Роскомнадзора ([rkn.gov.ru/communication/register/](https://rkn.gov.ru/communication/register/)).
2. **Хранение метаданных 3 года** (кто-кому-когда-откуда — IP, телефон,
   timestamp, факт соединения).
3. **Хранение контента 6 месяцев** — для мессенджера это `encrypted_content`.
   По факту E2E сервер отдаёт шифротекст — это допустимо, но шифротекст
   **обязан храниться**.
4. **Предоставление ФСБ информации, необходимой для декодирования**
   (Приказ ФСБ № 432, 374-ФЗ). Для честного E2E — физически невозможно, но
   писать «невозможно» нужно юридически корректно (см. [LEGAL_RESPONSE.md](LEGAL_RESPONSE.md)).
5. **Интеграция с СОРМ-3** — технические средства для органов. На практике
   для небольших сервисов РКН иногда не требует сразу, но формально обязан.
6. **Передача при запросах** — данных о пользователе, истории соединений,
   контента в зашифрованном виде.

**Санкции (КоАП 13.31)**:
- Нерегистрация: 300 000 – 800 000 ₽.
- Неисполнение обязанностей по хранению/передаче: 800 000 – 1 млн ₽.
- Повторно: до 6 млн ₽ + блокировка.

**Что это значит для Grizzly**:
- Нынешняя архитектура близка к соответствию (метаданные есть, шифротекст
  есть), но **нужен формализованный ретеншн-регламент** и **заполнение формы в
  реестре ОРИ**.
- **Без регистрации в ОРИ мессенджер запускать нельзя**. Это не «отложенное
  требование» — РКН сам приходит быстро, особенно для .ru-доменов.

### 2.3 🔴 374-ФЗ «пакет Яровой»

Применяется **к ОРИ из реестра** (п. 2.2). Ключевой момент:

- Метаданные — **3 года** (раньше говорил 1 год — ошибся).
- Контент сообщений — **6 месяцев**.
- Контент звонков — аудио/видео WebRTC **не проходит через сервер** (P2P через
  DTLS-SRTP или TURN-relay без записи), поэтому **хранить нечего** — это нужно
  зафиксировать в Privacy Policy и в технической справке для ФСБ.

### 2.4 🟡 ПП РФ № 313 — нотификация шифровальных средств

Форма: на сайте ЦЛСЗ ФСБ [clsz.fsb.ru](https://clsz.fsb.ru).

Заполняется по каждому шифровальному средству:
- **Крипта сообщений**: AES-256-GCM + X25519 ECDH + HKDF-SHA256 + PBKDF2 →
  одна нотификация, категория «массовое ПО», класс КС1 (без аппаратных
  средств защиты ключей — ключи в DataStore / Android Keystore).
- **WebRTC DTLS-SRTP** — отдельной нотификации не требует (это часть
  библиотеки `stream-webrtc-android`, которая уже использует стандартную
  TLS-стек Android, который ФСБ не считает подконтрольной крипто-разработкой
  оператора).
- **TLS транспорта** — не требует отдельной нотификации, т.к. вы не
  разрабатываете TLS, а используете OS-stack.

Срок: 10 рабочих дней, бесплатно. **Без нотификации продавать/распространять
нельзя**. Ответственность — ст. 13.12.1 КоАП (до 30 000 ₽ на должностное лицо,
до 300 000 ₽ на юрлицо) + конфискация.

### 2.5 🟡 152-ФЗ — персональные данные

**Что собирается**:
- Номер телефона (категория «общие ПДн»)
- Имя, аватар (общие ПДн)
- IP, геолокация сессии (технические ПДн)
- Метаданные сообщений (связаны с субъектом ПДн)

**Обязанности**:
1. **Уведомление РКН** об обработке ПДн — бесплатно, [pd.rkn.gov.ru](https://pd.rkn.gov.ru),
   онлайн, ~30 дней рассмотрения.
2. **Локализация** (ст. 18 ч. 5) — **первичная** запись ПДн россиян на серверах
   в РФ. Хостинг в Selectel / Yandex Cloud / VK Cloud устраивает.
3. **Privacy Policy** на [grizzly-messenger.ru](https://grizzly-messenger.ru)
   с перечислением всех категорий ПДн, целей, сроков хранения, третьих лиц.
4. **Согласие на обработку ПДн** при регистрации — явный чекбокс, не
   «молчаливое согласие».
5. **Назначение ответственного за обработку ПДн** (в ИП — сам ИП; в ООО —
   приказом).
6. **Политика в отношении обработки ПДн** — отдельный внутренний документ.
7. **Модель угроз безопасности ПДн** — обязательна с 2024 года (приказ
   ФСТЭК № 17).
8. **Оценка вреда субъектам ПДн** — новое с 2025 г.

**Санкции (КоАП 13.11, новая редакция 2025)**:
- Нарушения обработки: до 300 000 ₽.
- Утечка ПДн 1–10 тыс. субъектов: 3 000 000 – 5 000 000 ₽.
- Утечка 10–100 тыс.: 5 000 000 – 10 000 000 ₽.
- Утечка 100+ тыс. или спец. категорий: **до 3% выручки**.
- Повторная крупная утечка: до 500 000 000 ₽.

**⚠️ Самый дорогой риск на запуске**. Компрометация БД с номерами
телефонов 10k+ юзеров = штраф от 5 млн ₽ плюс уголовка по ст. 272 УК РФ
(неправомерный доступ к компьютерной информации) для ответственного.

### 2.6 🟡 SMS-рассылки OTP

Сейчас в плане Firebase/Twilio — **в РФ работать не будет** стабильно (Firebase
в РФ частично работает, Twilio маршрутизирует через серых агрегаторов, штрафы
за это).

Нужен договор с российским SMS-агрегатором, у которого есть лицензия на связь:
- SMS-Центр (smsc.ru) — самый распространённый
- SMS.ru
- Devino Telecom
- SigmaSMS

У каждого — договор с указанием назначения (OTP для мессенджера), согласование
имени отправителя (Grizzly), стоимость ~2–4 ₽/SMS.

Заодно этот же договор помогает с 241-ФЗ: агрегатор подтверждает, что OTP
отправлен на номер, зарегистрированный у оператора связи, но **полноценной
идентификацией** это **не является**. Для полной идентификации нужен отдельный
договор с оператором.

### 2.7 🟡 Закон о рекламе (38-ФЗ)

Если через Grizzly Bot отправляются коммерческие сообщения — каждое должно
помечаться «Реклама» + ERID (единый реестр интернет-рекламы, маркировка через
ОРД с 2022 года). Сообщение без маркировки — штраф 100 000 – 500 000 ₽ **за
каждое**.

Если бот шлёт только сервисные уведомления — это не реклама, маркировка не
нужна, но формулировки надо аккуратно подбирать («купите подписку» — реклама,
«ваша подписка заканчивается» — сервисное).

### 2.8 🟡 Распространение APK вне сторов

Формально не запрещено. Но:
- Пользователь должен в настройках включить «установку из неизвестных
  источников» — это фрикция.
- Google Play Protect показывает предупреждение.
- RuStore — лучший вариант. Там требование: подписать дистрибутивное
  соглашение с VK (как оператором RuStore), получить dev-аккаунт (бесплатно),
  пройти модерацию (~1 неделя).
- NashStore, AppGallery — опционально.

## 3. Что это значит для стадии «ИП нет, запуск только в РФ»

### Обязательно до публичного релиза

| № | Что | Срок | Стоимость | Блокер для запуска |
|---|---|---|---|---|
| 1 | Регистрация ИП (или ООО) | 3–7 дней | 0 ₽ (ИП) / 4000 ₽ (ООО) | ✅ |
| 2 | Нотификация ФСБ ЦЛСЗ | 10 дней | 0 ₽ | ✅ |
| 3 | Уведомление РКН об обработке ПДн | 30 дней | 0 ₽ | ✅ |
| 4 | Внесение в реестр ОРИ (РКН) | 30 дней | 0 ₽ | ✅ |
| 5 | Договор с SMS-агрегатором | 1–2 недели | 0–5 000 ₽ депозита | ✅ |
| 6 | Договор об идентификации с оператором связи (241-ФЗ) | 2–6 месяцев | от 50 000 ₽/мес | ✅ (без него — нельзя) |
| 7 | Privacy Policy + Пользовательское соглашение + Политика ПДн | 1–2 недели | 0 ₽ (шаблоны) или 30 000 ₽ (юрист) | ✅ |
| 8 | Модель угроз безопасности ПДн | 1 неделя | 0 ₽ (самостоятельно) или 50 000 ₽ (ИБ-консультант) | ✅ |
| 9 | Хостинг в РФ (Selectel / Yandex / VK Cloud) | 1 день | от 1 500 ₽/мес | ✅ |
| 10 | Opt-in-согласие на рассылки бота (фикс в коде) | 2–3 дня | dev-время | ✅ |
| 11 | Ретеншн-политика (cron-скрипты) | 3–5 дней | dev-время | ⚠️ Очень желательно |
| 12 | RuStore dev-аккаунт + публикация | 1 неделя | 0 ₽ | ⚠️ Желательно |

**Реалистичный срок до запуска**: 3–6 месяцев (лимитирующий фактор — пункт 6,
договор с оператором связи по 241-ФЗ).

**Минимальный бюджет на запуск**:
- ИП: бесплатно.
- Хостинг + SMS + оператор связи: ~60 000 ₽/мес постоянно.
- Юридическое сопровождение (документы, договоры): 50 000 – 150 000 ₽ разово.
- Адвокат на ретейнер (для [LEGAL_RESPONSE.md](LEGAL_RESPONSE.md)): 20 000 – 50 000 ₽/мес.
- **Итого старт**: ~200 000 ₽ + 80 000 ₽/мес.

### Когда нужен не ИП, а ООО

241-ФЗ не запрещает ИП быть организатором мессенджера, но **операторы связи
(МТС, Билайн, МегаФон) часто не заключают договор об идентификации с ИП** —
требуют юрлицо с уставным капиталом и ИБ-персоналом. На практике:
- Пробовать ИП с МВНО (тип Yota, Tinkoff Mobile) — иногда соглашаются.
- На ООО — почти всегда соглашаются.

Если цель — серьёзный публичный сервис → сразу **ООО на УСН 15%**.

## 4. Что в коде требует изменений до запуска

### 4.1 🔴 Критичные (без этого запуск = нарушение)

- [ ] **`GrizzlyBot` → opt-in-согласие** на получение сервисных/маркетинговых
  сообщений. Чекбокс в онбординге, запись в `users.bot_opt_in BOOL`.
- [ ] **241-ФЗ: блок неидентифицированных** — пока оператор связи не
  подтвердил номер, ограничивать функционал (например, нельзя писать, можно
  только читать или вообще ничего). Код проверки — на бэке, по результату
  вызова API оператора.
- [ ] **Чекбокс согласия на обработку ПДн** при регистрации, с текстом
  ссылкой на Privacy Policy. Сейчас в коде нужно проверить — см.
  [`app/src/main/java/com/secure/messenger/presentation/ui/auth/`](app/src/main/java/com/secure/messenger/presentation/ui/auth/).

### 4.2 🟡 Сильно желательные

- [ ] **PBKDF2 → Argon2id** или поднять итерации до 600 000+ для PBKDF2-SHA256.
  Сейчас 100k — в 2026 году слабовато для офлайн-брутфорса.
- [ ] **Хеширование `users.phone`** (SHA-256 с pepper) при хранении.
  Использовать хеш для lookup, сырой номер только в момент отправки SMS.
  План есть в [LEGAL_RESPONSE.md:155](LEGAL_RESPONSE.md#L155), не реализован.

## 7. Альтернативы, если полный РФ-комплаенс неподъёмен

### 7.1 Позиционирование как «внутренний инструмент»

Не публичный мессенджер, а **B2B-продукт для закрытых команд**. Компания
покупает развёртывание Grizzly для своих сотрудников, каждый развёртывает у
себя, вы — вендор ПО, а не оператор сервиса.

- **241-ФЗ не применяется** (нет публичного сервиса).
- **ОРИ не применяется** (оператор — конечный клиент).
- Нотификация ФСБ — остаётся.
- 152-ФЗ на клиенте, не на вас.

Бизнес-модель: лицензии / подписка на обновления / кастомизация.

### 7.2 Запуск вне РФ

Серверы в нейтральной юрисдикции (Нидерланды, Швейцария, Казахстан),
приложение в Google Play на международном рынке. Не скачивается из РФ официально —
регуляторы РФ формально не достают. Аудитория россиян получает через VPN.
- Плюсы: 0 комплаенса с РФ, честный E2E как фича.
- Минусы: возможна блокировка по IP через РКН, нет монетизации в РФ-сторах,
  никакой работы с РФ-бизнесом.

### 7.3 Open-source без оператора

Опубликовать на GitHub как OSS, без «грей», без домена, без сервера в
продакшене. Каждый разворачивает сам. Обязательств нет.
- Минусы: нет монетизации, проект живёт только пока вы его поддерживаете.

## 8. Главное в одном абзаце

Первая оценка была оптимистичной: «ИП + нотификация ФСБ + РКН + хостинг в РФ»
— это минимум для **любого** интернет-сервиса с ПДн, но **не для мессенджера**.
Grizzly по факту кода — это одновременно ОРИ (149-ФЗ), мессенджер по 241-ФЗ и
оператор ПДн по 152-ФЗ. Критический путь — **договор с оператором связи об
идентификации** (2–6 месяцев). Без него мессенджер не может легально работать в
РФ, и это не «отложенный риск», а блокер запуска. Плюс функция `GrizzlyBot`
(массовая рассылка админом) в нынешней реализации — прямое нарушение 241-ФЗ
без opt-in-согласия пользователей. Криптография в коде — корректная, стандартная,
проходит нотификацию без проблем. Суммарный срок до легального публичного
запуска — 3–6 месяцев, бюджет — ~200 000 ₽ разово + ~80 000 ₽/мес. Если это
неподъёмно — смотреть альтернативы из раздела 7 (B2B, зарубежный запуск,
open-source).
